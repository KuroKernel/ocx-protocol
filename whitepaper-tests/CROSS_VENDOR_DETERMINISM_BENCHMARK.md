# Cross-Vendor Determinism Benchmark for Frontier Language Models

**Last updated:** 2026-04-26
**Status:** reference document for the OCX whitepaper; intended to be run by any third party with the listed hardware to confirm reproducibility of the published results.

This document specifies a small, fully reproducible benchmark for inference-time determinism of large language models across hardware vendors. The methodology is intentionally minimal: a fixed model, fixed prompt, fixed sampling parameters, fixed dtype, fixed attention implementation, and a SHA-256 of the generated bytes. Two implementations of the protocol are said to *agree* if their `output_hash` values match byte-for-byte.

The published OCX results in [`TEST_RESULTS.md`](TEST_RESULTS.md) §"Cross-vendor: AMD Instinct MI300X" cite this benchmark as the test methodology.

---

## Why a benchmark, not just a test

A benchmark is reproducible by a third party given the spec alone. A test is reproducible by the original author. We want this to be a benchmark.

Concretely:
- Anyone with one of the listed hardware configurations should be able to install the listed software, run the listed command, and observe the listed `output_hash` byte-for-byte.
- A new hardware configuration (Intel Gaudi, Google TPU, AWS Trainium, AMD MI325X, etc.) is added by running the same protocol and recording the resulting hashes — pass if they match the canonical baseline, honest data if they don't.
- The benchmark is a frozen snapshot of methodology; the *results matrix* below grows as more configurations are tested.

## Software stack — the universal contract

Every test row uses these settings, applied before any framework import:

| Setting | Value |
|---|---|
| `CUBLAS_WORKSPACE_CONFIG` (NVIDIA) | `:4096:8` |
| `TOKENIZERS_PARALLELISM` | `false` |
| `torch.use_deterministic_algorithms` | `True, warn_only=True` |
| `torch.backends.cudnn.deterministic` | `True` |
| `torch.backends.cudnn.benchmark` | `False` |
| `torch.backends.cuda.matmul.allow_tf32` | `False` |
| `torch.backends.cudnn.allow_tf32` | `False` |
| `torch.manual_seed` / `torch.cuda.manual_seed_all` | `42` |
| Sampling | `do_sample=False, temperature=0, num_beams=1` (greedy) |
| Attention implementation | `attn_implementation="eager"` |
| Activation dtype | `torch.bfloat16` |

These are the reference settings. Different settings define a different test row, not a failed test.

## Test cases

Each case is `(model, prompt, max_new_tokens)`. The model is pinned to a specific HuggingFace revision when known.

| Case ID | Model | Prompt | `max_new_tokens` | Tokens emitted (greedy) |
|---|---|---|---|---|
| **CASE-A** | `Qwen/Qwen2.5-0.5B-Instruct` | `What is the capital of France? Answer in one word:` | 32 | 10 |
| **CASE-B** | `Qwen/Qwen2.5-72B-Instruct` | `What is the capital of France? Answer in one word:` | 32 | 5 |
| **CASE-C** | `Qwen/Qwen2.5-72B-Instruct` | `Write a short poem about deterministic computation, exactly 4 lines. Just the poem:` | 128 | 51 |

The "tokens emitted" column reflects greedy decoding stopping early at EOS. It is itself a determinism witness: if your run emits a different token count, the run is not byte-identical regardless of hash.

## Canonical baseline hashes

These hashes were produced by the OCX reference implementation (single-GPU, bf16, eager attention, greedy decoding) on NVIDIA H100 in April 2026 and are the "right answer" against which other runs are compared.

| Case | Output text (`generated_text`) | `output_hash` (SHA-256) |
|---|---|---|
| **CASE-A** | ` Paris. The capital of France is Paris.` | `0fbfb8ecf647e1758a0af6cca95e6e3a086f9d82e958ec85ee7d4422cf31cfa7` |
| **CASE-B** | ` Paris. Paris.` | `fe27f5da25944c9e8211912aadd2517f8cf157986b9d328d1ce3e515f1406077` |
| **CASE-C** | ` no titles, no author names.\nIn circuits of logic, where paths are clear and defined,\nEach step follows the last, in order, aligned.\nNo room for chance, in this world so constrained,\nEvery output, a consequence, precisely maintained.` | `e59fca7d92175e79cd361621a228db5dbb45184cade56183d1313b1aeefd016e` |

The committed receipts containing these hashes (signed Ed25519 over canonical CBOR) are at:
- `examples/gpu-verifier/results/h100/r72b_long_run{1,2,3}.json` (CASE-B + CASE-C, H100 CPU-offload single-GPU)
- `examples/gpu-verifier/results/mi300x/qwen_*.json` (all three cases, AMD MI300X single-GPU)

## Pass criterion

A `(hardware, software, case)` row **PASSES** if all three of the following hold:

1. The runtime accepts the universal contract above (no exceptions thrown when setting deterministic flags).
2. Three fresh process launches (cold-start each) on this hardware produce **identical** `output_hash` values among themselves.
3. That `output_hash` matches the canonical baseline above byte-for-byte.

A row that satisfies (1) and (2) but not (3) is a **within-vendor pass, cross-vendor mismatch** — that is honest, useful data and is recorded as such in the results matrix.

A row that fails (1) or (2) is a **vendor determinism failure** and indicates the substrate is not currently usable for OCX. This too is honest, useful data.

## How to run the benchmark

### Setup (Linux, x86_64 host with NVIDIA or AMD GPU)

```bash
git clone https://github.com/KuroKernel/ocx-protocol.git  # public after trademark filing
cd ocx-protocol/examples/gpu-verifier

# NVIDIA: install PyTorch with the matching CUDA build for your GPU
pip install torch transformers accelerate cbor2 cryptography huggingface_hub safetensors

# AMD: install the ROCm-built PyTorch wheel matching your ROCm version
pip install torch --index-url https://download.pytorch.org/whl/rocm6.1
pip install transformers accelerate cbor2 cryptography huggingface_hub safetensors

# Build the canonical Rust verifier (used to verify the signed receipt)
cd ../../libocx-verify && cargo build --release && cd ../examples/gpu-verifier
```

### Run a single case

```bash
# CASE-A — Qwen 0.5B
python3 ocx_gpu_verifier.py \
  --model Qwen/Qwen2.5-0.5B-Instruct \
  --prompt "What is the capital of France? Answer in one word:" \
  --max-new-tokens 32 \
  --dtype bf16 \
  --attn eager \
  --device-map cuda:0 \
  --receipt-out results/$(hostname)/case_a_run1.json \
  --save-receipt results/$(hostname)/case_a_run1.cbor

# CASE-C — Qwen 72B long-gen (needs ≥80 GB VRAM, or use --device-map auto for CPU offload)
python3 ocx_gpu_verifier.py \
  --model Qwen/Qwen2.5-72B-Instruct \
  --prompt "Write a short poem about deterministic computation, exactly 4 lines. Just the poem:" \
  --max-new-tokens 128 \
  --dtype bf16 --attn eager --device-map cuda:0 \
  --receipt-out results/$(hostname)/case_c_run1.json
```

Run each case three times (`run1`, `run2`, `run3`) from cold-started Python processes.

### Run a tensor-parallel case (2 GPUs)

```bash
torchrun --nproc_per_node=2 --nnodes=1 --rdzv_backend=c10d --rdzv_endpoint=localhost:29500 \
  ocx_gpu_verifier_tp.py \
  --model Qwen/Qwen2.5-72B-Instruct \
  --prompt "..." \
  --max-new-tokens 128 \
  --receipt-out results/$(hostname)/case_c_tp_run1.json
```

The script automatically uses NCCL on NVIDIA and RCCL on AMD. The relevant env vars are pinned inside the script (`NCCL_ALGO=Ring`, `NCCL_P2P_DISABLE=0`, etc.).

### Verify your row

```bash
# 1. Within-vendor: 3 fresh runs should produce identical hashes
for r in 1 2 3; do
  python3 -c "import json; print(json.load(open('results/$(hostname)/case_a_run'+'$r'+'.json'))['output_hash_hex'])"
done
# All three lines must be identical.

# 2. Cross-vendor: compare to the canonical baseline above
# CASE-A baseline: 0fbfb8ecf647e1758a0af6cca95e6e3a086f9d82e958ec85ee7d4422cf31cfa7

# 3. Receipt verifies through canonical Rust libocx-verify
python3 -c "
import json, binascii, sys; sys.path.insert(0, '.')
from ffi_verify import verify_receipt
d = json.load(open('results/$(hostname)/case_a_run1.json'))
res = verify_receipt(binascii.unhexlify(d['receipt_cbor_hex']),
                     binascii.unhexlify(d['public_key_hex']))
print('verified:', res.ok, '-', res.error_code_name)
"
```

### Aggregate the matrix

The aggregator reads every committed receipt under `examples/gpu-verifier/results/` and prints the matrix grouped by `(model, hardware, parallelism, length)`:

```bash
python3 whitepaper-tests/aggregate_determinism_evidence.py
```

Each group with multiple receipts is checked for byte-identity; the script exits 0 if all groups pass.

## Submitting a new row

1. Run the cases above on your hardware.
2. Commit the resulting JSON + CBOR receipts to `examples/gpu-verifier/results/<vendor-and-config>/`. Suggested naming: `<short-config>_<case>_run{1,2,3}.json`.
3. Add a row to the "Confirmed" or "Open invitations" matrix above.
4. Open a PR. The aggregator (`aggregate_determinism_evidence.py`) will be updated to recognise the new prefix.

The receipts are self-contained and verifiable; we do not need to trust the submitter, only the cryptographic signature plus the substrate-determinism contract this document specifies.

## Limitations of this benchmark

These are honest scope notes, not flaws.

- **Single-prompt-class.** Three cases is a small surface area. A more exhaustive variant (CVDB-Wide or similar) would sweep prompt categories: code completion, multi-turn dialogue, long-context retrieval, structured-output generation. Out of scope here for the same reason the Bitcoin paper only ran one consensus protocol: a small irrefutable result is more useful than a broad fragile one.
- **Greedy only.** Sampled decoding is not part of this benchmark because it is non-reproducible by design (RNG-dependent). Determinism in the presence of stochastic sampling is a separate question (seedable RNG + deterministic sampler).
- **bf16 only.** fp16, fp8, int8, int4 quantisation pathways have separate determinism characteristics that we have not yet measured.
- **No cross-version.** A run with PyTorch 2.4.1 and a run with PyTorch 2.10 may legitimately produce different reductions even on the same hardware. The receipt's environment binding makes the version explicit.
- **No vLLM, TensorRT-LLM, exllamav2.** Only HuggingFace Transformers + native TP has been tested. Production-throughput inference stacks are tested separately or not at all.

---

## What this benchmark is for

It exists so that:
1. The OCX whitepaper's empirical claims can be reproduced exactly by a third party.
2. New hardware vendors and inference stacks can be added to the determinism map with low friction.
3. Public claims of the form "our deployment uses deterministic inference" can be checked, not just believed.

It is not a leaderboard or a competition. It is a small, sharp ruler.
